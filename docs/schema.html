Database Schema

SportDeck uses 7 MySQL tables. All foreign keys use ON DELETE CASCADE to keep data consistent.

Overview

# Table Description Key Relationships
1 users All accounts (admin & player) —
2 tournaments Tournament listings —
3 teams Teams per tournament → tournaments
4 players Player roster per team → teams, users
5 matches Scheduled matches → tournaments, teams
6 results Match outcomes & scores → matches, teams
7 tournament_registrations Player self-registrations → users, tournaments

users

Stores all user accounts. Both admins and players share this table, differentiated by the role column.

ColumnTypeNullableNotes
idINTNoPrimary key, auto-increment
nameVARCHAR(100)NoFull name
emailVARCHAR(100)NoUnique — used for login
password_hashVARCHAR(255)NoPHP password_hash() output
ageINTYesOptional
contactVARCHAR(20)YesPhone number
roleENUMNo'player' (default) or 'admin'
created_atTIMESTAMPNoAuto-set on insert

tournaments

Each row represents one tournament. The status field drives what actions players and admins can take.

ColumnTypeNullableNotes
idINTNoPrimary key
nameVARCHAR(200)NoTournament title
sport_typeVARCHAR(50)Noe.g. Basketball, Soccer
start_dateDATENo—
end_dateDATENo—
statusENUMNoupcoming · ongoing · completed
max_playersINTYesNULL = unlimited

teams

Teams belong to a specific tournament. Deleting a tournament cascades to all its teams.

ColumnTypeNullableNotes
idINTNoPrimary key
nameVARCHAR(100)NoTeam name
coach_nameVARCHAR(100)YesOptional
tournament_idINTNoFK → tournaments.id

players

Player records are part of team rosters. Optionally linked to a users account via user_id.

ColumnTypeNullableNotes
idINTNoPrimary key
nameVARCHAR(100)NoPlayer name
ageINTYes—
contactVARCHAR(20)Yes—
team_idINTNoFK → teams.id
user_idINTYesFK → users.id (SET NULL on delete)

matches

Each match is between two teams in the same tournament. A result row is created separately after completion.

ColumnTypeNullableNotes
idINTNoPrimary key
tournament_idINTNoFK → tournaments.id
team1_idINTNoFK → teams.id
team2_idINTNoFK → teams.id
match_dateDATETIMENoDate and time of match
venueVARCHAR(100)YesLocation name
statusENUMNoupcoming or completed

results

One result per match (enforced via UNIQUE on match_id). winner_team_id is NULL for draws.

ColumnTypeNullableNotes
idINTNoPrimary key
match_idINTNoFK → matches.id · UNIQUE
winner_team_idINTYesNULL = draw · FK → teams.id
scoreVARCHAR(50)Noe.g. 3-1
created_atTIMESTAMPNoAuto-set on insert

tournament_registrations

Tracks player self-registrations. A unique constraint prevents duplicate entries per user + tournament pair.

ColumnTypeNullableNotes
idINTNoPrimary key
user_idINTNoFK → users.id
tournament_idINTNoFK → tournaments.id
registered_atTIMESTAMPNoAuto-set on insert
statusENUMNoregistered (default) or cancelled
ℹ️ A UNIQUE KEY unique_reg (user_id, tournament_id) prevents a player from registering for the same tournament twice.

Relationships

Here's a summary of how all tables relate to each other:

users ──────────────────────────────────────────┐
  │                                             │
  │ (user_id, optional)                         │ (user_id)
  ▼                                             ▼
players ◄──── teams ◄──── tournaments ────► tournament_registrations
                │               │
                │               └──────────────────► matches
                │                                        │
                └─────────── (team1_id / team2_id) ──────┘
                                                         │
                                                         ▼
                                                      results
ℹ️ All foreign keys use ON DELETE CASCADE except players.user_id which uses SET NULL, so deleting a user account doesn't remove their roster entry.
